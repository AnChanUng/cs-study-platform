## 질문
대칭키 암호화와 공개키(비대칭키) 암호화의 차이점을 설명해주세요.

## 핵심 키워드
- 대칭키 (Symmetric Key)
- 공개키/개인키 (Public/Private Key)
- 키 교환 문제
- AES / DES
- RSA

## 답변

### 대칭키 암호화 (Symmetric Key Encryption)

암호화와 복호화에 동일한 키를 사용하는 방식입니다.

```
[평문] → 대칭키로 암호화 → [암호문] → 같은 대칭키로 복호화 → [평문]

송신자: "Hello" + Key → "x7f9a..."
수신자: "x7f9a..." + Key → "Hello"
```

**대표 알고리즘**: AES, DES, 3DES, SEED

**장점:**
- 암호화/복호화 속도가 빠름
- 대용량 데이터 처리에 적합

**단점:**
- **키 배포 문제**: 상대방에게 키를 안전하게 전달하는 것이 어려움
- 통신 상대가 많을수록 관리할 키가 급증 (n명이면 n(n-1)/2개의 키 필요)

### 공개키(비대칭키) 암호화 (Asymmetric Key Encryption)

암호화와 복호화에 서로 다른 키(공개키/개인키)를 사용합니다. 공개키로 암호화한 데이터는 개인키로만 복호화할 수 있습니다.

```
[평문] → 수신자의 공개키로 암호화 → [암호문] → 수신자의 개인키로 복호화 → [평문]

A→B: "Hello" + B의 공개키 → "k3m8..." → B의 개인키 → "Hello"
```

**대표 알고리즘**: RSA, ECC, ElGamal

**장점:**
- 키 배포 문제 해결 (공개키는 누구에게나 공개 가능)
- 디지털 서명에 활용 가능

**단점:**
- 암호화/복호화 속도가 느림 (대칭키 대비 약 1000배)
- 대용량 데이터 처리에 부적합

### 디지털 서명

개인키로 서명하고, 공개키로 검증합니다.

```
서명: 원본 데이터의 해시 → 송신자의 개인키로 암호화 = 서명
검증: 서명 → 송신자의 공개키로 복호화 = 해시 → 원본 해시와 비교
```

이를 통해 **인증**(누가 보냈는지)과 **무결성**(변조되지 않았는지)을 보장합니다.

### 비교

| 구분 | 대칭키 | 비대칭키 |
|------|--------|---------|
| 키 수 | 1개 (공유) | 2개 (공개키 + 개인키) |
| 속도 | 빠름 | 느림 |
| 키 배포 | 어려움 | 쉬움 |
| 키 관리 | n(n-1)/2개 | 2n개 |
| 용도 | 데이터 암호화 | 키 교환, 디지털 서명 |
| 대표 알고리즘 | AES, DES | RSA, ECC |

### 하이브리드 방식 (실무)

HTTPS/TLS에서는 두 방식을 결합하여 사용합니다:
1. 비대칭키로 대칭키를 안전하게 교환
2. 교환된 대칭키로 실제 데이터를 암호화

```
1. 서버가 공개키 전송
2. 클라이언트가 대칭키 생성
3. 공개키로 대칭키를 암호화하여 서버에 전송
4. 이후 대칭키로 데이터 암호화 통신
```

### 면접 팁
실무에서는 두 방식을 결합한 하이브리드 방식을 사용한다는 점이 핵심입니다. HTTPS에서 어떻게 적용되는지 구체적으로 설명할 수 있으면 좋습니다.

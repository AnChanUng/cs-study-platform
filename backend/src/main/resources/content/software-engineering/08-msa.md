## 답변
마이크로서비스 아키텍처(MSA)는 하나의 애플리케이션을 여러 개의 작고 독립적인 서비스로 분리하여 개발하고 배포하는 아키텍처 스타일입니다.

### 모놀리식 vs 마이크로서비스

```
모놀리식:                마이크로서비스:
┌─────────────┐         ┌────┐ ┌────┐ ┌────┐
│  사용자 관리  │         │사용자│ │주문 │ │결제 │
│  주문 관리   │         │서비스│ │서비스│ │서비스│
│  결제 관리   │         └──┬─┘ └──┬─┘ └──┬─┘
│  알림 관리   │            │      │      │
│  단일 DB    │         ┌──┴─┐ ┌──┴─┐ ┌──┴─┐
└─────────────┘         │DB  │ │DB  │ │DB  │
                        └────┘ └────┘ └────┘
```

| 구분 | 모놀리식 | MSA |
|------|---------|-----|
| 배포 | 전체 배포 | 서비스별 독립 배포 |
| 확장 | 전체 확장 | 서비스별 독립 확장 |
| 기술 스택 | 단일 | 서비스별 선택 가능 |
| 데이터베이스 | 공유 DB | 서비스별 독립 DB |
| 장애 영향 | 전체 장애 | 해당 서비스만 |
| 복잡도 | 낮음 | 높음 (분산 시스템) |

### MSA 핵심 구성요소

**1. API Gateway**
클라이언트 요청을 적절한 서비스로 라우팅합니다.
- 인증/인가
- 로드 밸런싱
- 요청/응답 변환
- 속도 제한 (Rate Limiting)

**2. 서비스 간 통신**
- **동기 통신**: REST API, gRPC
- **비동기 통신**: 메시지 큐 (Kafka, RabbitMQ)

```java
// REST 동기 호출
@FeignClient(name = "order-service")
public interface OrderClient {
    @GetMapping("/orders/{userId}")
    List<Order> getOrders(@PathVariable Long userId);
}

// Kafka 비동기 메시지
@KafkaListener(topics = "order-created")
public void handleOrderCreated(OrderEvent event) {
    paymentService.processPayment(event);
}
```

**3. 서비스 디스커버리**
서비스의 위치(IP, 포트)를 동적으로 관리합니다. (Eureka, Consul)

**4. 서킷 브레이커**
장애가 발생한 서비스로의 호출을 차단하여 장애 전파를 방지합니다. (Resilience4j)

### MSA의 장점
- 서비스별 독립적 개발, 배포, 확장
- 기술 스택의 자유도
- 장애 격리
- 팀별 자율적 개발

### MSA의 단점
- 분산 시스템의 복잡성 (네트워크 지연, 장애)
- 분산 트랜잭션 처리의 어려움
- 서비스 간 통신 오버헤드
- 운영 복잡도 증가 (모니터링, 로깅)
- 데이터 일관성 유지 어려움

### 면접 팁
MSA를 무조건 좋다고 하지 말고, 트레이드오프를 명확히 설명하세요. 작은 팀이나 초기 프로젝트에서는 모놀리식이 더 적합할 수 있습니다. "모놀리식 먼저, 필요에 따라 MSA로 전환"이 일반적인 접근법입니다.

## 질문
B-Tree와 B+Tree의 구조와 차이점에 대해 설명하고, 데이터베이스에서 B+Tree를 사용하는 이유를 설명해주세요.

## 핵심 키워드
- 다진 탐색 트리
- 균형 트리
- 디스크 I/O 최소화
- 범위 탐색
- 데이터베이스 인덱스

## 답변
B-Tree와 B+Tree는 디스크 기반의 대용량 데이터를 효율적으로 관리하기 위해 설계된 균형 다진 탐색 트리(Balanced Multi-way Search Tree)입니다.

### B-Tree

차수(order) M인 B-Tree의 특성:
- 루트 노드는 최소 2개의 자식을 가짐
- 루트를 제외한 내부 노드는 최소 ⌈M/2⌉개의 자식을 가짐
- 모든 리프 노드는 같은 레벨에 위치 (균형)
- 각 노드는 최대 M-1개의 키를 가짐
- 키는 정렬된 상태로 유지

```
B-Tree (차수 3):
        [30]
       /    \
   [10, 20]  [40, 50]
   / | \      / | \
  ... ... ... ... ... ...
```

**장점:**
- 모든 노드에 데이터가 저장되어 있어 검색 시 빨리 찾을 수 있음
- 디스크 I/O를 최소화하도록 설계됨 (높이가 낮음)

### B+Tree

B-Tree의 변형으로, 데이터베이스 인덱스에 가장 널리 사용됩니다.

**B-Tree와의 차이점:**
1. **데이터 저장 위치**: 실제 데이터는 리프 노드에만 저장되고, 내부 노드는 키만 저장 (인덱스 역할)
2. **리프 노드 연결**: 리프 노드들이 연결 리스트로 연결되어 있음
3. **키 중복**: 내부 노드의 키가 리프 노드에도 존재

```
B+Tree (차수 3):
         [30]              ← 내부 노드 (키만 저장)
        /    \
   [10, 20]  [30, 40]     ← 내부 노드 (키만 저장)
   / | \      / | \
  [데이터들]              ← 리프 노드 (실제 데이터)
  ←→ ←→ ←→ ←→ ←→         ← 리프 노드 간 연결 리스트
```

### 비교

| 특성 | B-Tree | B+Tree |
|------|--------|--------|
| 데이터 저장 | 모든 노드 | 리프 노드만 |
| 리프 연결 | 없음 | 연결 리스트 |
| 내부 노드 크기 | 큼 | 작음 (키만) |
| 범위 탐색 | 비효율적 | 효율적 |
| 중복 키 | 없음 | 있음 |

### B+Tree가 DB 인덱스에 사용되는 이유

**1. 범위 탐색 효율성**
리프 노드가 연결 리스트로 연결되어 있어 순차 탐색이 매우 효율적입니다.
```sql
-- B+Tree가 유리한 범위 쿼리
SELECT * FROM employees WHERE age BETWEEN 25 AND 35;
```

**2. 내부 노드의 작은 크기**
내부 노드에 데이터 없이 키만 저장하므로 하나의 노드에 더 많은 키를 담을 수 있습니다. 이는 트리의 높이를 낮추어 디스크 I/O 횟수를 줄입니다.

**3. 균일한 검색 성능**
모든 데이터가 리프 노드에 있으므로, 어떤 데이터를 검색하든 동일한 시간이 소요됩니다.

### 시간 복잡도
- 검색: O(log n)
- 삽입: O(log n)
- 삭제: O(log n)

### 면접 팁
B+Tree가 데이터베이스 인덱스에 사용되는 이유를 디스크 I/O 관점에서 설명할 수 있어야 합니다. 해시 인덱스와 비교하여 B+Tree가 범위 탐색에서 유리한 이유도 함께 설명하세요.

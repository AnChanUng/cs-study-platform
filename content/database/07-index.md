## 질문
데이터베이스 인덱스(Index)란 무엇이며, 어떻게 동작하나요?

## 핵심 키워드
- B-Tree 인덱스
- 클러스터드 인덱스 / 논클러스터드 인덱스
- 해시 인덱스
- 인덱스 스캔 vs 풀 스캔
- 카디널리티

## 답변
인덱스(Index)는 데이터베이스 테이블에 대한 검색 속도를 향상시키기 위한 자료구조입니다. 책의 색인(목차)처럼 원하는 데이터의 위치를 빠르게 찾을 수 있게 해줍니다.

### 동작 원리

인덱스가 없으면 테이블 전체를 순차적으로 탐색(Full Table Scan)해야 합니다. 인덱스가 있으면 B-Tree 등의 자료구조를 통해 O(log N) 시간에 데이터를 찾을 수 있습니다.

```sql
-- 인덱스 생성
CREATE INDEX idx_employee_name ON employee(name);

-- 인덱스를 활용한 조회
SELECT * FROM employee WHERE name = '홍길동';
```

### 인덱스 종류

**1. B-Tree 인덱스**
가장 일반적인 인덱스 구조입니다. 범위 검색, 정렬에 유리합니다. 리프 노드에 실제 데이터의 포인터가 저장됩니다.

**2. 해시 인덱스**
해시 함수를 사용하여 O(1) 시간에 검색합니다. 동등 비교(=)에는 빠르지만 범위 검색(>, <)에는 사용할 수 없습니다.

**3. 클러스터드 인덱스 (Clustered Index)**
테이블의 물리적 정렬 순서와 인덱스 순서가 동일합니다. 테이블당 하나만 생성 가능하며, 보통 기본키에 자동으로 생성됩니다.

**4. 논클러스터드 인덱스 (Non-clustered Index)**
물리적 정렬과 독립적으로 별도의 공간에 인덱스를 저장합니다. 테이블당 여러 개 생성 가능합니다.

### 인덱스 사용 시 고려사항

**인덱스가 효과적인 경우:**
- WHERE 절에 자주 사용되는 컬럼
- JOIN에 사용되는 컬럼
- ORDER BY, GROUP BY에 사용되는 컬럼
- 카디널리티(Cardinality)가 높은 컬럼 (고유한 값이 많은 컬럼)

**인덱스가 비효율적인 경우:**
- 데이터가 적은 테이블
- INSERT, UPDATE, DELETE가 빈번한 테이블
- 카디널리티가 낮은 컬럼 (성별 등)

### 인덱스의 단점
- 추가적인 저장 공간이 필요합니다 (테이블 크기의 약 10%)
- 데이터 변경(INSERT, UPDATE, DELETE) 시 인덱스도 함께 갱신되어 성능이 저하됩니다
- 잘못된 인덱스는 오히려 성능을 악화시킬 수 있습니다

```sql
-- 실행 계획으로 인덱스 사용 여부 확인
EXPLAIN SELECT * FROM employee WHERE name = '홍길동';
```

### 면접 팁
인덱스의 내부 구조(B-Tree)를 이해하고, 인덱스를 사용해야 할 때와 사용하지 않아야 할 때를 구분할 수 있어야 합니다. 커버링 인덱스, 복합 인덱스의 컬럼 순서 등 심화 주제도 준비하세요.

## 질문
로드 밸런싱이란 무엇이며, 어떤 알고리즘들이 있나요?

## 핵심 키워드
- 부하 분산
- L4 / L7 로드 밸런서
- 라운드 로빈 / 가중치
- 헬스 체크
- 세션 유지

## 답변
로드 밸런싱(Load Balancing)은 들어오는 네트워크 트래픽을 여러 서버에 분산시켜 시스템의 가용성과 성능을 높이는 기술입니다.

### 로드 밸런서 종류

**L4 로드 밸런서 (전송 계층)**
- IP 주소와 포트 번호를 기반으로 트래픽 분산
- 패킷의 내용을 확인하지 않아 속도가 빠름
- 비교적 간단한 분산만 가능

**L7 로드 밸런서 (응용 계층)**
- HTTP 헤더, URL, 쿠키 등을 분석하여 트래픽 분산
- 콘텐츠 기반의 지능적인 분산 가능
- SSL 종료(SSL Termination) 처리 가능

```
예: L7 로드 밸런서
/api/*     → API 서버 그룹
/images/*  → 이미지 서버 그룹
/static/*  → CDN
```

### 로드 밸런싱 알고리즘

**1. 라운드 로빈 (Round Robin)**
요청을 순서대로 각 서버에 분배합니다.
```
요청 1 → 서버 A
요청 2 → 서버 B
요청 3 → 서버 C
요청 4 → 서버 A (다시 처음부터)
```

**2. 가중치 라운드 로빈 (Weighted Round Robin)**
서버의 처리 능력에 따라 가중치를 부여합니다.
```
서버 A (가중치 5): 5개 요청 처리
서버 B (가중치 3): 3개 요청 처리
서버 C (가중치 2): 2개 요청 처리
```

**3. 최소 연결 (Least Connection)**
현재 연결 수가 가장 적은 서버에 요청을 전달합니다. 세션이 긴 요청에 적합합니다.

**4. IP 해시 (IP Hash)**
클라이언트 IP를 해시하여 항상 같은 서버에 연결합니다. 세션 유지에 유리합니다.

**5. 최소 응답 시간 (Least Response Time)**
응답 시간이 가장 짧고 연결 수가 적은 서버를 선택합니다.

### 헬스 체크 (Health Check)

로드 밸런서는 주기적으로 서버의 상태를 확인합니다.
```
- TCP 연결 확인
- HTTP 요청 전송 후 상태 코드 확인 (200 OK)
- 특정 URL 응답 확인 (/health 엔드포인트)

장애 서버 감지 → 트래픽 분배에서 제외
복구 감지 → 트래픽 분배에 다시 포함
```

### 세션 유지 (Session Persistence)

로드 밸런싱 환경에서 같은 사용자의 요청이 다른 서버로 분산되면 세션 문제가 발생합니다.

**해결 방법:**
1. **Sticky Session**: 특정 사용자를 항상 같은 서버에 연결
2. **세션 클러스터링**: 서버 간 세션 공유 (Redis 등)
3. **JWT 토큰**: Stateless 인증으로 세션 문제 회피

### 실무 구성 예시
```
                     ┌─── 서버 A
사용자 → CDN → L7 LB ├─── 서버 B
                     └─── 서버 C

대표 제품: AWS ALB/NLB, Nginx, HAProxy
```

### 면접 팁
L4와 L7의 차이, 알고리즘별 적합한 상황, 세션 유지 문제와 해결 방법을 함께 설명하세요. 실무에서 Nginx나 AWS ELB를 사용한 경험이 있다면 함께 언급하면 좋습니다.

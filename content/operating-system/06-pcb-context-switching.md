## 질문
PCB(Process Control Block)와 Context Switching에 대해 설명해주세요.

## 핵심 키워드
- PCB (프로세스 제어 블록)
- Context Switching (문맥 교환)
- 프로세스 상태 정보
- 레지스터 저장/복원
- 오버헤드

## 답변

### PCB (Process Control Block)
PCB는 운영체제가 각 프로세스를 관리하기 위해 유지하는 자료구조입니다. 프로세스가 생성되면 PCB가 만들어지고, 종료되면 제거됩니다.

**PCB에 저장되는 정보:**

```
┌─────────────────────────┐
│ 프로세스 ID (PID)        │
│ 프로세스 상태             │ (Ready, Running, Waiting 등)
│ 프로그램 카운터 (PC)      │ 다음 실행할 명령어 주소
│ CPU 레지스터 정보         │ 레지스터 값들
│ CPU 스케줄링 정보         │ 우선순위, 스케줄링 큐 포인터
│ 메모리 관리 정보          │ 페이지 테이블, 세그먼트 테이블
│ I/O 상태 정보            │ 할당된 I/O 장치, 열린 파일 목록
│ 계정 정보                │ CPU 사용 시간, 시간 제한
└─────────────────────────┘
```

### Context Switching (문맥 교환)

현재 실행 중인 프로세스(또는 스레드)의 상태를 PCB에 저장하고, 다음 실행할 프로세스의 상태를 PCB에서 복원하여 CPU를 할당하는 과정입니다.

**발생 시점:**
- 프로세스의 타임 슬라이스가 만료되었을 때
- I/O 요청으로 프로세스가 대기 상태로 전환될 때
- 더 높은 우선순위의 프로세스가 도착했을 때
- 인터럽트가 발생했을 때

**Context Switching 과정:**

```
[프로세스 A 실행 중]
    ↓ 인터럽트 또는 시스템 콜
[프로세스 A의 상태를 PCB_A에 저장]
    - PC, 레지스터, 스택 포인터 등
    ↓
[스케줄러가 다음 프로세스 B를 선택]
    ↓
[PCB_B에서 프로세스 B의 상태를 복원]
    - PC, 레지스터, 스택 포인터 등
    ↓
[프로세스 B 실행 시작]
```

### Context Switching 오버헤드

Context Switching 자체는 유용한 작업을 수행하지 않는 순수한 오버헤드입니다.

**오버헤드 원인:**
1. PCB 저장 및 복원에 소요되는 시간
2. CPU 캐시(Cache) 무효화 - 새 프로세스의 데이터로 캐시를 다시 채워야 함
3. TLB(Translation Lookaside Buffer) 플러시
4. 파이프라인 플러시

**프로세스 vs 스레드의 Context Switching:**
- **프로세스**: 메모리 주소 공간이 다르므로 페이지 테이블, TLB 등도 교체 → 비용 큼
- **스레드**: 같은 프로세스 내 스레드는 메모리를 공유하므로 → 비용 작음

### 오버헤드 최소화 방법
- 스레드 사용 (메모리 공유로 비용 감소)
- 적절한 스케줄링 알고리즘 선택
- 너무 잦은 Context Switching 방지 (타임 슬라이스 조절)

### 면접 팁
Context Switching의 비용이 왜 발생하는지(캐시 무효화, TLB 플러시)를 구체적으로 설명할 수 있으면 좋습니다. 프로세스 간 Context Switching이 스레드 간보다 비용이 큰 이유를 메모리 구조의 차이와 연결하여 설명하세요.
